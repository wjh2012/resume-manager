코드 변경사항을 분석하고 docs/ 문서를 동기화하세요.

## 작업 절차

1. 아래 두 범위를 합산하여 변경된 코드 파일 확인 (인자가 있으면 해당 범위 사용)
   - **커밋된 변경**: `git diff --name-only @{upstream}..HEAD` (upstream 미설정 시 `origin/{현재브랜치}..HEAD`)
   - **아직 커밋 안 한 변경**: `git diff --name-only` (unstaged) + `git diff --name-only --staged` (staged)
2. 변경된 코드 파일을 읽고 내용 분석
3. `docs/` 하위 전체 문서 목록을 확인하고, 변경 내용이 영향을 주는 문서 식별
4. 해당 문서를 읽고 코드와 불일치하는 부분 확인
5. 문서 내용을 코드에 맞게 갱신
6. 핵심 기능이 새로 추가된 경우 `docs/features/`에 기능 문서 작성
7. 변경 없으면 "동기화 불필요" 보고

## docs/ 디렉토리 구조

| 디렉토리 | 용도 |
|---|---|
| `docs/features/` | 핵심 기능별 설명 문서 (기능 목적, 동작 방식, 주요 컴포넌트, 데이터 흐름) |
| `docs/rules/` | 코드 컨벤션, 아키텍처 규칙, 작업 프로세스, 커뮤니케이션 규칙 |
| `docs/guides/` | 기술 가이드, 설정 가이드 |
| `docs/changelog/` | 대규모 변경 이력 |
| `docs/references/` | 프로젝트 참고 자료 (known-issues, spec-deviations 등) |

## 동기화 대상 판단

변경된 코드를 분석하여 `docs/` 하위 **모든 문서**를 대상으로 관련성을 판단한다. 고정 매핑이 아닌, 변경 내용과 문서 내용을 비교하여 동적으로 판단한다.

### docs/features/ 문서 작성 기준

**새로 작성**: 핵심 기능이 신규 추가되었을 때
**갱신**: 기존 기능의 동작 방식, 컴포넌트 구성, 데이터 흐름이 변경되었을 때
**불필요**: 내부 리팩토링, 버그 수정, 스타일링만 변경

#### features 문서 형식

```markdown
# {기능명}

## 개요
기능의 목적과 사용자 가치를 1-2문장으로 설명.

## 동작 방식
핵심 흐름을 단계별로 설명.

## 주요 컴포넌트
기능을 구성하는 파일/모듈과 각 역할.

## 데이터 흐름
입력 → 처리 → 출력 흐름. 필요 시 상태 관리 포함.
```

### 일반 문서 갱신 판단 기준

**필요**: 필드/타입 변경, API 추가/삭제, 설정값 변경, 패키지 구조 변경, 새 패턴 도입, 워크플로우 변경
**불필요**: 내부 리팩토링, 버그 수정, 테스트만 변경, 포맷팅만 변경

## 규칙

- 코드에 실제로 반영된 패턴만 문서에 기록
- 기존 문서 구조/형식 유지
- 불필요한 내용 추가 금지
- 확실하지 않은 내용은 코드를 직접 읽어서 확인
- 문서가 없는 디렉토리는 필요 시 생성

## 출력 형식

```
## 동기화 결과

### 신규 작성
- `docs/features/xxx.md` — 기능 설명 요약

### 갱신됨
- `docs/rules/xxx.md` — 변경 내용 요약

### 동기화 불필요
- `docs/workflow/yyy.md` — 사유
```
