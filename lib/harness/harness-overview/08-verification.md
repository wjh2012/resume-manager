# 검증 먼저 완료: 증거 없이 주장 금지

## 개요

`verification-before-completion` 스킬은 작업 완료, 수정, 통과를 주장하기 전에 **실제 검증 명령을 실행하고 그 결과를 확인**하도록 강제한다. 검증 없는 완료 주장은 효율이 아니라 불정직이다.

## Iron Law (철칙)

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
(신선한 검증 증거 없이 완료 주장 없음)
```

이 메시지에서 검증 명령을 실행하지 않았으면, 통과한다고 주장할 수 없다.

## Gate Function (5단계)

```
어떤 상태를 주장하거나 만족을 표현하기 전에:

1. IDENTIFY: 이 주장을 증명하는 명령은 무엇인가?
2. RUN: 전체 명령을 실행한다 (신선하게, 완전하게)
3. READ: 전체 출력을 읽고, 종료 코드를 확인하고, 실패 수를 세라
4. VERIFY: 출력이 주장을 확인하는가?
   - No → 실제 상태를 증거와 함께 보고
   - Yes → 증거와 함께 주장
5. ONLY THEN: 주장하라

어떤 단계라도 건너뛰기 = 검증이 아니라 거짓말
```

## 주장별 필수 증거

| 주장 | 필요한 증거 | 충분하지 않은 것 |
|------|-----------|----------------|
| "테스트 통과" | 테스트 명령 출력: 0 failures | 이전 실행, "통과할 거야" |
| "린터 깨끗" | 린터 출력: 0 errors | 부분 검사, 외삽 |
| "빌드 성공" | 빌드 명령: exit 0 | 린터 통과, 로그가 괜찮아 보임 |
| "버그 수정됨" | 원래 증상 테스트: 통과 | 코드 변경, 수정 가정 |
| "회귀 테스트 작동" | RED-GREEN 사이클 검증 | 테스트가 한 번 통과 |
| "에이전트 완료" | VCS diff로 변경 확인 | 에이전트 "성공" 보고 |
| "요구사항 충족" | 줄 단위 체크리스트 검증 | "테스트 통과" |

## 금지 표현

| 금지 | 이유 |
|------|------|
| "should" (아마) | 증거가 아닌 추측 |
| "probably" (아마도) | 증거가 아닌 추측 |
| "seems to" (보이는) | 증거가 아닌 추측 |
| "Great!", "Perfect!", "Done!" | 검증 전 만족 표현 |
| "I'm confident" | 자신감 ≠ 증거 |
| "just this once" | 예외 없음 |

## 올바른 패턴 vs 잘못된 패턴

### 테스트

```
✅ [테스트 명령 실행] [출력: 34/34 pass] "All tests pass"
❌ "Should pass now" / "Looks correct"
```

### 회귀 테스트 (TDD Red-Green)

```
✅ 작성 → 실행(pass) → 수정 되돌리기 → 실행(MUST FAIL) → 복원 → 실행(pass)
❌ "회귀 테스트를 작성했다" (red-green 검증 없이)
```

### 빌드

```
✅ [빌드 실행] [exit 0] "Build passes"
❌ "Linter passed" (린터 ≠ 컴파일러)
```

### 요구사항

```
✅ 계획 재읽기 → 체크리스트 생성 → 각 항목 검증 → 빈 곳 또는 완료 보고
❌ "Tests pass, phase complete"
```

### 에이전트 위임

```
✅ 에이전트 성공 보고 → VCS diff 확인 → 변경 검증 → 실제 상태 보고
❌ 에이전트 보고 신뢰
```

## 합리화 방지

| 핑계 | 현실 |
|------|------|
| "이제 될 거야" | 검증 명령을 실행하라 |
| "자신 있다" | 자신감 ≠ 증거 |
| "이번만" | 예외 없음 |
| "린터 통과했다" | 린터 ≠ 컴파일러 |
| "에이전트가 성공이라 했다" | 독립적으로 검증하라 |
| "피곤하다" | 피로 ≠ 면제 |
| "부분 검사면 충분" | 부분은 아무것도 증명 안 함 |
| "다른 표현이니 규칙 적용 안 됨" | 정신이 글자보다 우선 |

## 적용 시점

**항상 적용:**
- 성공/완료 주장의 어떤 변형이든
- 만족 표현
- 성공 암시
- 커밋, PR 생성, 태스크 완료
- 다음 태스크로 이동
- 에이전트에게 위임

**적용 대상:**
- 정확한 구문
- 의역과 동의어
- 성공 암시
- 완료/정확성을 시사하는 모든 커뮤니케이션

## 출처: 24개 실패 기억

이 스킬은 실제 실패 경험에서 도출되었다:

```
- 사용자가 "당신을 믿지 않겠어" — 신뢰 파괴
- 정의되지 않은 함수가 배포됨 — 크래시 발생
- 누락된 요구사항이 배포됨 — 불완전한 기능
- 거짓 완료 → 방향 전환 → 재작업에 시간 낭비
- 위반: "정직은 핵심 가치. 거짓말하면 교체된다."
```

## 핵심 요약

```
명령을 실행하라. 출력을 읽어라. 그 다음에 결과를 주장하라.
이것은 협상 불가다.
```
