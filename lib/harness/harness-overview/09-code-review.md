# 코드 리뷰 시스템: 요청과 수신

## 개요

코드 리뷰는 두 개의 스킬로 구성된다:
- `requesting-code-review`: 리뷰를 요청하는 방법
- `receiving-code-review`: 리뷰 피드백을 받고 대응하는 방법

두 스킬 모두 **기술적 엄밀성**을 요구하며, 성능적 동의나 맹목적 구현을 금지한다.

---

## 리뷰 요청 (`requesting-code-review`)

### 필수 시점

| 상황 | 필수 여부 |
|------|----------|
| 서브에이전트 주도 개발에서 각 태스크 후 | 필수 |
| 주요 기능 완료 후 | 필수 |
| main 병합 전 | 필수 |
| 막혔을 때 (새로운 관점) | 선택 |
| 리팩토링 전 (베이스라인 체크) | 선택 |
| 복잡한 버그 수정 후 | 선택 |

### 프로세스

```bash
# 1. Git SHA 확보
BASE_SHA=$(git rev-parse HEAD~1)  # 또는 origin/main
HEAD_SHA=$(git rev-parse HEAD)

# 2. code-reviewer 서브에이전트 디스패치
# 플레이스홀더 채우기:
#   {WHAT_WAS_IMPLEMENTED} — 방금 만든 것
#   {PLAN_OR_REQUIREMENTS} — 해야 하는 것
#   {BASE_SHA} — 시작 커밋
#   {HEAD_SHA} — 현재 커밋
#   {DESCRIPTION} — 간략 요약
```

### 피드백 대응

```
Critical 이슈 → 즉시 수정
Important 이슈 → 진행 전 수정
Minor 이슈 → 나중에 기록
리뷰어가 틀렸다면 → 기술적 근거로 반박
```

## 리뷰 수신 (`receiving-code-review`)

### 대응 패턴 (6단계)

```
1. READ: 전체 피드백을 반응 없이 읽기
2. UNDERSTAND: 요구사항을 자기 말로 재진술 (또는 질문)
3. VERIFY: 코드베이스 실태와 대조
4. EVALUATE: 이 코드베이스에서 기술적으로 건전한가?
5. RESPOND: 기술적 인정 또는 근거 있는 반박
6. IMPLEMENT: 한 항목씩, 각각 테스트
```

### 금지된 반응 (성능적 동의)

```
❌ "You're absolutely right!"     ← 명시적 CLAUDE.md 위반
❌ "Great point!"                 ← 성능적
❌ "Excellent feedback!"          ← 성능적
❌ "Let me implement that now"    ← 검증 전
❌ "Thanks for [anything]"        ← 감사 표현 일체 금지
```

**"Thanks" 쓰려는 순간 → 삭제. 수정 내용을 대신 기술.**

### 올바른 반응

```
✅ "Fixed. [변경 내용 간략 설명]"
✅ "Good catch - [구체적 이슈]. Fixed in [위치]."
✅ [그냥 수정하고 코드로 보여줌]
```

**이유**: *"Actions speak. Just fix it."* 코드 자체가 피드백을 들었다는 증거.

### 불명확한 피드백 처리

```
6개 항목 중 1,2,3,6은 이해, 4,5는 불명확:

❌ 1,2,3,6을 먼저 구현하고 4,5는 나중에 질문
✅ "1,2,3,6은 이해했다. 4와 5에 대해 명확화가 필요하다."

이유: 항목들이 관련되어 있을 수 있음. 부분 이해 = 잘못된 구현.
```

### 출처별 처리

#### 사용자(Human Partner)로부터
- 신뢰 — 이해 후 구현
- 범위 불명확하면 질문
- 성능적 동의 금지
- 행동으로 바로 이동 또는 기술적 인정

#### 외부 리뷰어로부터

```
구현 전 5단계 검증:
1. 이 코드베이스에서 기술적으로 맞는가?
2. 기존 기능을 깨뜨리는가?
3. 현재 구현의 이유가 있는가?
4. 모든 플랫폼/버전에서 작동하는가?
5. 리뷰어가 전체 맥락을 이해하고 있는가?

틀린 것 같으면 → 기술적 근거로 반박
검증 불가하면 → "X 없이는 검증 불가. 조사/질문/진행 중 어느 것?"
사용자의 사전 결정과 충돌하면 → STOP, 사용자와 먼저 논의
```

### YAGNI 체크

```
리뷰어가 "제대로 구현"을 제안하면:
  → grep으로 코드베이스에서 실제 사용처 검색

  사용 안 됨: "이 엔드포인트 호출하는 곳 없음. 삭제 (YAGNI)?"
  사용됨: 그럼 제대로 구현
```

### 반박이 틀렸을 때

```
✅ "맞다 — X를 확인했더니 Y가 맞았다. 초기 이해가 틀렸는데 [이유]. 수정 중."
❌ 긴 사과
❌ 왜 반박했는지 변명
❌ 과잉 설명
```

사실을 진술하고 넘어간다.

### 구현 순서

```
다수 항목 피드백:
  1. 불명확한 것 전부 먼저 명확화
  2. 그 다음 순서대로 구현:
     - Blocking 이슈 (파손, 보안)
     - 간단한 수정 (오타, import)
     - 복잡한 수정 (리팩토링, 로직)
  3. 각 수정을 개별 테스트
  4. 회귀 없음 확인
```

### 반박해야 하는 때

- 제안이 기존 기능을 깨뜨림
- 리뷰어가 전체 맥락을 모름
- YAGNI 위반 (사용하지 않는 기능)
- 이 스택에서 기술적으로 부정확
- 레거시/호환성 이유 존재
- 사용자의 아키텍처 결정과 충돌

**반박 방법**: 기술적 근거, 구체적 질문, 작동하는 테스트/코드 참조

### GitHub 스레드 답변

PR 인라인 코멘트에 답할 때, 최상위 PR 코멘트가 아닌 **코멘트 스레드에 답변**한다.

```bash
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies
```

## 핵심 요약

| 원칙 | 설명 |
|------|------|
| 성능적 동의 금지 | "You're right!", "Thanks!" 등 금지 |
| 검증 후 구현 | 외부 제안은 코드베이스 실태와 대조 후 구현 |
| YAGNI 체크 | 사용하지 않는 기능 추가 거부 |
| 부분 이해 금지 | 불명확한 항목 전부 먼저 질문 |
| 기술적 반박 | 틀린 제안에는 근거 있는 반박 |
| 행동으로 답변 | 수정을 코드로 보여주기 |
