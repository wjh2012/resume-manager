# 브레인스토밍: 설계 선행 원칙

## 개요

`brainstorming` 스킬은 모든 창의적 작업(기능 생성, 컴포넌트 빌드, 기능 추가, 행동 변경) 전에 반드시 거쳐야 하는 **설계 단계**이다. 아이디어를 자연스러운 협업 대화를 통해 완성된 설계와 스펙으로 변환한다.

## HARD-GATE 메커니즘

```xml
<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project,
or take any implementation action until you have presented a design and the
user has approved it.
</HARD-GATE>
```

XML 태그를 사용한 이유: 일반 마크다운보다 LLM이 구조적 지시를 더 강하게 인식한다. `<HARD-GATE>`는 "넘을 수 없는 문턱"이라는 의미를 전달한다.

## 안티패턴: "설계가 필요 없을 만큼 단순하다"

```
모든 프로젝트가 이 프로세스를 거친다.
TODO 리스트, 단일 함수 유틸리티, 설정 변경 — 전부.
설계가 짧을 수 있지만(몇 문장), 반드시 제시하고 승인받아야 한다.

"Simple" projects are where unexamined assumptions cause the most wasted work.
```

## 9단계 체크리스트

각 단계는 순서 의존적이며, 건너뛰기가 구조적으로 불가능하다:

### 1. 프로젝트 컨텍스트 탐색
파일, 문서, 최근 커밋을 확인하여 현재 프로젝트 상태를 파악한다.

### 2. 비주얼 컴패니언 제안
시각적 질문이 예상되면 **별도 메시지**로 제안한다. 다른 내용(질문, 컨텍스트 요약 등)과 **절대 결합하지 않는다**.

```
"Some of what we're working on might be easier to explain if I can show it
to you in a web browser. Want to try it?"
```

사용자가 수락해도 **매 질문마다** 브라우저/터미널을 개별 판단한다:
- 브라우저: 목업, 와이어프레임, 레이아웃 비교, 아키텍처 다이어그램
- 터미널: 요구사항 질문, 개념적 선택, 트레이드오프, 범위 결정

### 3. 명확화 질문
- 한 번에 **하나의 질문**만
- 가능하면 **객관식** 선호
- 목적, 제약, 성공 기준 이해에 집중

### 4. 2-3개 접근법 제안
- 트레이드오프와 추천 포함
- 추천 옵션을 먼저 제시하고 이유 설명

### 5. 설계 제시
- 복잡도에 비례한 섹션별 제시 (단순하면 몇 문장, 복잡하면 200-300단어)
- 각 섹션 후 "여기까지 괜찮은가?" 확인
- 커버 범위: 아키텍처, 컴포넌트, 데이터 플로우, 에러 처리, 테스팅

### 6. 설계 문서 작성
- `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`에 저장
- git에 커밋

### 7. 스펙 셀프 리뷰
4가지 점검:
1. **Placeholder 스캔**: "TBD", "TODO", 불완전 섹션?
2. **내부 일관성**: 섹션 간 모순? 아키텍처와 기능 설명 일치?
3. **범위 확인**: 단일 구현 계획에 적합한 범위인가?
4. **모호성 확인**: 두 가지로 해석 가능한 요구사항? → 하나를 선택하고 명시

### 8. 사용자 스펙 리뷰
```
"Spec written and committed to <path>.
Please review it and let me know if you want to make any changes
before we start writing out the implementation plan."
```
사용자 승인까지 대기. 변경 요청 시 수정 후 셀프 리뷰 재실행.

### 9. 구현 전환
`writing-plans` 스킬 호출. **다른 어떤 스킬도 호출하지 않는다.**

## 설계 원칙: 격리와 명확성

```
시스템을 더 작은 단위로 분해한다. 각 단위는:
- 하나의 명확한 목적
- 잘 정의된 인터페이스로 소통
- 독립적으로 이해하고 테스트 가능

자기 검증 질문:
- "내부를 읽지 않고도 무엇을 하는지 이해할 수 있는가?"
- "소비자를 깨뜨리지 않고 내부를 바꿀 수 있는가?"
- 아니면, 경계를 다시 잡아야 한다.
```

**실용적 근거**: 에이전트는 컨텍스트에 담을 수 있는 코드에 대해 더 잘 추론하고, 파일이 집중적일 때 편집이 더 신뢰성 있다.

## 대규모 프로젝트 분해

```
요청이 다수의 독립 서브시스템을 기술하면 (예: "채팅, 파일 스토리지, 빌링, 분석이 있는 플랫폼"):
  1. 즉시 분해 필요성을 지적
  2. 독립적 조각, 관계, 빌드 순서 정리
  3. 첫 번째 서브 프로젝트를 정상 설계 플로우로 브레인스토밍
  4. 각 서브 프로젝트가 자체 스펙 → 계획 → 구현 사이클을 가짐
```

## 기존 코드베이스에서의 작업

```
- 변경 제안 전에 현재 구조를 먼저 탐색
- 기존 패턴을 따름
- 작업에 영향을 미치는 기존 문제가 있으면 (파일 비대, 불명확 경계) 설계에 개선 포함
- 관련 없는 리팩토링은 제안하지 않음
```

## 종단 상태 잠금

```
브레인스토밍의 유일한 출구 → writing-plans 호출
frontend-design, mcp-builder 등 구현 스킬 직접 호출 금지
```

이것은 파이프라인 순서를 구조적으로 강제한다.

## 핵심 원칙 요약

| 원칙 | 설명 |
|------|------|
| HARD-GATE | 설계 승인 전 코드 작성 금지 |
| "단순해도 설계" | 모든 프로젝트에 적용, 예외 없음 |
| 한 번에 하나의 질문 | 사용자를 압도하지 않음 |
| YAGNI | 불필요한 기능 무자비하게 제거 |
| 점진적 검증 | 섹션별 제시 → 승인 → 다음 |
| 종단 상태 잠금 | writing-plans만 호출 가능 |
