# 체계적 디버깅: 4단계 과학적 방법

## 개요

`systematic-debugging` 스킬은 버그, 테스트 실패, 예상치 못한 행동을 만났을 때 **근본 원인을 먼저 찾도록** 강제한다. 임의 수정은 시간 낭비와 새 버그를 만든다.

## Iron Law (철칙)

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
(근본 원인 조사 없이 수정 없음)
```

Phase 1을 완료하지 않았으면 수정을 **제안조차** 할 수 없다.

## 적용 범위

모든 기술적 이슈에 적용:
- 테스트 실패, 프로덕션 버그, 예상치 못한 행동
- 성능 문제, 빌드 실패, 통합 이슈

**특히 적용해야 하는 상황:**
- 시간 압박 하에서 (긴급 상황이 추측을 유혹)
- "빠르게 하나만 수정"이 명백해 보일 때
- 이미 여러 수정을 시도했을 때
- 이전 수정이 작동하지 않았을 때

## 4단계 프로세스

### Phase 1: 근본 원인 조사 (수정 시도 전 필수)

**1. 에러 메시지 꼼꼼히 읽기**
- 스택 트레이스 완전히 읽기
- 라인 번호, 파일 경로, 에러 코드 기록
- 에러 메시지에 정확한 해법이 포함된 경우가 많음

**2. 일관되게 재현**
- 정확한 단계는?
- 매번 발생하는가?
- 재현 불가 → 더 많은 데이터 수집 (추측 금지)

**3. 최근 변경 확인**
- git diff, 최근 커밋
- 새 의존성, 설정 변경
- 환경 차이

**4. 다중 컴포넌트 증거 수집**

시스템이 여러 컴포넌트를 가질 때, 수정 전에 **각 컴포넌트 경계에 진단 계측을 추가**한다:

```bash
# 계층 1: 워크플로우
echo "=== Secrets available: ==="
echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# 계층 2: 빌드 스크립트
echo "=== Env vars in build: ==="
env | grep IDENTITY

# 계층 3: 서명 스크립트
echo "=== Keychain state: ==="
security list-keychains

# 계층 4: 실제 서명
codesign --sign "$IDENTITY" --verbose=4 "$APP"
```

한 번 실행하여 **어디서 깨지는지** 증거를 수집한 후 분석.

**5. 데이터 흐름 추적**
- 나쁜 값이 **어디서 시작되는가?**
- 무엇이 나쁜 값으로 이것을 호출했는가?
- 원점을 찾을 때까지 역방향 추적
- **원점에서 수정** (증상에서 수정하지 않음)

### Phase 2: 패턴 분석

**1. 동작하는 유사 코드 찾기**
- 같은 코드베이스에서 동작하는 유사한 것은?

**2. 참조 구현 비교**
- 참조 구현을 **완전히** 읽기 (훑어보기 금지)
- 패턴을 완전히 이해한 후 적용

**3. 차이점 식별**
- 동작하는 것과 깨진 것 사이의 모든 차이점 나열
- "그건 상관없다"고 가정하지 않기

**4. 의존성 이해**
- 필요한 다른 컴포넌트는?
- 설정, 환경?
- 암묵적 가정은?

### Phase 3: 가설과 테스트

**과학적 방법 적용:**

1. **단일 가설 형성**: "X가 근본 원인이라고 생각한다, 왜냐하면 Y"
2. **최소 변경으로 테스트**: 한 번에 하나의 변수만 변경
3. **검증**:
   - 작동? → Phase 4
   - 미작동? → 새로운 가설 (기존 수정 위에 더 쌓지 않음)
4. **모를 때**: "X를 이해하지 못한다"고 인정. 추가 조사 또는 도움 요청.

### Phase 4: 구현

**1. 실패하는 테스트 케이스 생성**
- 가장 간단한 재현
- `superpowers:test-driven-development` 스킬 사용
- 수정 전에 반드시 존재해야 함

**2. 단일 수정 구현**
- 식별된 근본 원인만 해결
- 한 번에 하나의 변경만
- "하는 김에" 개선 금지
- 번들 리팩토링 금지

**3. 수정 검증**
- 테스트 통과?
- 다른 테스트 깨지지 않음?
- 이슈가 실제로 해결됨?

**4. 수정이 작동하지 않으면**
- **시도 횟수 카운트**
- 3회 미만: Phase 1로 돌아가 새 정보로 재분석
- **3회 이상: STOP — Phase 4.5로**

### Phase 4.5: 아키텍처 의심 (3회 이상 실패 시)

```
패턴 신호:
  - 각 수정이 다른 곳에서 새로운 공유 상태/결합 문제를 드러냄
  - 수정에 "대규모 리팩토링"이 필요
  - 각 수정이 다른 곳에서 새 증상을 만듦

이것은 실패한 가설이 아니라 잘못된 아키텍처.

근본을 의심:
  - 이 패턴이 근본적으로 건전한가?
  - "관성으로 고집하고 있는 건 아닌가?"
  - 증상 수정 대신 아키텍처를 리팩토링해야 하나?

사용자와 논의 없이 Fix #4 시도 금지.
```

## 사용자 신호 인식

사용자의 특정 표현이 "접근이 잘못되었다"는 신호:

| 사용자 표현 | 의미 |
|-------------|------|
| "그게 안 일어나나?" | 검증 없이 가정했음 |
| "그게 보여줄까...?" | 증거 수집을 추가했어야 함 |
| "추측 그만해" | 이해 없이 수정 제안 중 |
| "Ultrathink해" | 증상이 아닌 근본을 의심하라 |
| "막혔나?" (짜증) | 접근이 작동하지 않음 |

**이 신호를 보면: STOP. Phase 1로 돌아감.**

## Red Flags — 멈추고 프로세스 따르기

```
- "빠른 수정하고, 나중에 조사"
- "X를 바꿔보고 되는지 보자"
- "여러 변경 추가하고 테스트 실행"
- "테스트 건너뛰고 수동 확인"
- "아마 X일 거야, 수정하자"
- "완전히 이해하지 못하지만 될 수도"
- "주요 문제 목록" (조사 없이 수정 나열)
- "한 번 더 수정 시도" (이미 2회 이상)
```

**전부: STOP. Phase 1로 돌아감.**

## 합리화 방지

| 핑계 | 현실 |
|------|------|
| "이슈가 간단해서 프로세스 필요 없음" | 간단한 이슈에도 근본 원인 있음. 프로세스는 빠름 |
| "긴급, 프로세스 할 시간 없음" | 체계적 디버깅이 추측보다 빠름 |
| "이것만 먼저 시도하고 조사" | 첫 수정이 패턴을 설정. 처음부터 올바르게 |
| "여러 수정 한 번에 하면 시간 절약" | 무엇이 작동했는지 분리 불가. 새 버그 발생 |
| "한 번 더 시도" (2회+ 실패 후) | 3회+ 실패 = 아키텍처 문제. 패턴 의심 |

## 실제 영향

```
체계적 접근: 15-30분에 수정
임의 수정 접근: 2-3시간 삽질
첫 시도 수정 성공률: 95% vs 40%
도입된 새 버그: 거의 없음 vs 흔함
```
