# 서브에이전트 주도 개발: 실행 엔진

## 개요

`subagent-driven-development`는 전체 하네스의 **실행 엔진**이다. 계획을 태스크 단위로 분해하고, 각 태스크에 fresh 서브에이전트를 디스패치하여 구현한 후, 2단계 리뷰(스펙 준수 → 코드 품질)를 거친다.

## 핵심 원리

```
태스크당 새 서브에이전트 + 2단계 리뷰(스펙 → 품질) = 고품질, 빠른 반복
```

**왜 서브에이전트인가:**
- 격리된 컨텍스트 → 컨텍스트 오염 방지
- 정확히 필요한 정보만 구성하여 제공 → 집중도 보장
- 세션 히스토리를 상속하지 않음 → 순수한 태스크 수행
- 컨트롤러의 컨텍스트 윈도우 보존 → 조율 작업 유지

## 사용 조건

```
구현 계획이 있는가?
  └── Yes → 태스크가 대부분 독립적인가?
       └── Yes → 현재 세션에서 작업?
            └── Yes → subagent-driven-development
            └── No → executing-plans (병렬 세션)
       └── No (밀접 결합) → 수동 실행 또는 브레인스토밍
  └── No → 브레인스토밍 먼저
```

## 전체 실행 루프

```
[계획 읽기] → [모든 태스크 전체 텍스트 추출] → [TodoWrite 생성]

각 태스크에 대해:
  ┌─────────────────────────────────────────────────┐
  │ 1. Implementer 서브에이전트 디스패치              │
  │    ├── 질문 있음? → 답변 + 재디스패치             │
  │    └── 질문 없음? → 구현 + 테스트 + 커밋 + 셀프리뷰│
  │                                                   │
  │ 2. Spec Reviewer 서브에이전트 디스패치             │
  │    ├── ❌ 이슈 → Implementer가 수정 → 재리뷰      │
  │    └── ✅ 스펙 준수 → 다음 단계                   │
  │                                                   │
  │ 3. Code Quality Reviewer 서브에이전트 디스패치     │
  │    ├── ❌ 이슈 → Implementer가 수정 → 재리뷰      │
  │    └── ✅ 승인 → 태스크 완료 표시                  │
  └─────────────────────────────────────────────────┘

모든 태스크 완료 → 최종 코드 리뷰어 디스패치 → finishing-a-development-branch
```

## 컨트롤러-서브에이전트 분리

### 컨트롤러 (메인 에이전트)
- 계획 파일을 **한 번** 읽고 모든 태스크의 전체 텍스트 추출
- 각 서브에이전트에 정확한 컨텍스트를 구성하여 제공
- 조율(coordination)만 수행, 직접 구현하지 않음

### 서브에이전트
- 자신의 태스크만 알고 있음
- **계획 파일을 직접 읽지 않음** (컨트롤러가 전체 텍스트 제공)
- 세션 히스토리를 상속하지 않음

## 세 가지 역할

### Implementer (`implementer-prompt.md`)

완전한 태스크 텍스트 + 컨텍스트를 받아 구현한다.

**프롬프트 구조:**
1. **Task Description** — 계획에서 추출한 전체 텍스트
2. **Context** — 이 태스크가 전체에서 어디에 위치하는지
3. **Before You Begin** — 작업 전 질문 기회
4. **Your Job** — 구현 → 테스트 → 검증 → 커밋 → 셀프리뷰
5. **Code Organization** — 파일 책임, 크기 주의
6. **When You're in Over Your Head** — 에스컬레이션 허가
7. **Self-Review** — 완전성, 품질, 규율, 테스팅 4축 검토
8. **Report Format** — 4가지 상태 보고

**에스컬레이션 허가:**
```
"이건 나한테 너무 어렵다"고 멈추고 말하는 것은 항상 OK.
나쁜 작업은 작업을 안 하는 것보다 나쁘다.
에스컬레이션에 페널티는 없다.

STOP 조건:
- 여러 유효한 접근법의 아키텍처 결정이 필요
- 제공된 것 너머의 코드 이해가 필요
- 접근의 정확성이 불확실
- 계획이 예상하지 못한 기존 코드 구조 변경 필요
```

### Spec Reviewer (`spec-reviewer-prompt.md`)

**핵심 지시: 의도적 불신**

```
구현자가 의심스러울 정도로 빨리 완료했다.
보고서가 불완전하거나, 부정확하거나, 낙관적일 수 있다.
모든 것을 독립적으로 검증해야 한다.

하지 마라:
  - 구현한 것에 대한 구현자의 말을 믿지 마라
  - 완전성 주장을 믿지 마라
  - 요구사항 해석을 받아들이지 마라

해라:
  - 실제 코드를 읽어라
  - 요구사항과 줄 단위로 비교하라
  - 구현했다고 주장하나 실제로 빠진 부분을 찾아라
  - 언급하지 않은 추가 기능을 찾아라
```

**검증 범주:**
- **Missing**: 누락된 요구사항
- **Extra**: 요청하지 않은 추가 기능
- **Misunderstanding**: 잘못된 해석

### Code Quality Reviewer (`code-quality-reviewer-prompt.md`)

**스펙 리뷰 통과 후에만 디스패치** (순서 강제).

추가 점검 사항:
- 각 파일이 하나의 명확한 책임을 가지는가?
- 독립적으로 이해하고 테스트할 수 있는 단위인가?
- 계획의 파일 구조를 따르는가?
- 이번 변경으로 새로 만들어진 파일이 이미 큰가?

**출력 형식**: Strengths, Issues (Critical/Important/Minor), Assessment

## 리뷰 순서 강제

```
스펙 준수 리뷰 ✅ → 코드 품질 리뷰

스펙 준수 전에 코드 품질 리뷰 시작 = Red Flag
```

**이유**: 잘못된 것을 아름답게 만드는 것을 방지. 먼저 "맞는지" → 그 다음 "잘 만들었는지".

## Implementer 상태 처리 (4가지)

| 상태 | 의미 | 대응 |
|------|------|------|
| **DONE** | 완료 | 스펙 리뷰 진행 |
| **DONE_WITH_CONCERNS** | 완료했으나 의문 | 우려 읽기 → 정확성/범위면 해결, 관찰이면 기록 후 진행 |
| **NEEDS_CONTEXT** | 정보 부족 | 누락 컨텍스트 제공 + 재디스패치 |
| **BLOCKED** | 수행 불가 | 단계적 확대 (아래 참조) |

### BLOCKED 대응 — 단계적 확대

```
1. 컨텍스트 문제 → 더 많은 컨텍스트 + 같은 모델로 재디스패치
2. 추론 부족 → 더 유능한 모델로 재디스패치
3. 태스크 과대 → 더 작은 조각으로 분해
4. 계획 자체 오류 → 사용자에게 에스컬레이션
```

**절대 하지 말 것**: 에스컬레이션을 무시하거나 변경 없이 같은 모델로 재시도.

## 모델 선택 전략

```
기계적 구현 (격리된 함수, 명확한 스펙, 1-2 파일):
  → 빠르고 저렴한 모델
  → 잘 명세된 계획이 있으면 대부분 기계적

통합/판단 태스크 (다중 파일, 패턴 매칭, 디버깅):
  → 표준 모델

아키텍처/설계/리뷰:
  → 가장 유능한 모델
```

## Red Flags (하지 말 것)

- main/master 브랜치에서 구현 시작 (사용자 동의 없이)
- 리뷰 건너뛰기 (스펙 준수 또는 코드 품질)
- 미해결 이슈로 진행
- 다수 구현 서브에이전트 병렬 디스패치 (충돌 발생)
- 서브에이전트가 계획 파일을 읽게 하기 (전체 텍스트 제공)
- 장면 설정 컨텍스트 생략
- 서브에이전트 질문 무시
- 스펙 준수에서 "거의 맞음" 수용
- 리뷰 루프 생략 (이슈 발견 → 수정 → 재리뷰 → 승인까지)
- 셀프 리뷰가 실제 리뷰를 대체하게 하기

## 이점 요약

| 측면 | 이점 |
|------|------|
| 컨텍스트 | 태스크당 fresh 컨텍스트 (오염 없음) |
| 품질 | 3단계 검증 (셀프리뷰 + 스펙 + 품질) |
| 비용 | 조기 이슈 포착 (나중에 디버깅보다 저렴) |
| 속도 | 지속적 진행 (사람 개입 없이 태스크 간 이동) |
| 안전 | 병렬 안전 (서브에이전트 간 간섭 없음) |
