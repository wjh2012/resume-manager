## 8. 결론 및 권장사항

### 프로덕션 적용

**S6 (S5 + Pre-check)를 프로덕션에 적용하는 것을 권장한다.**

1. **시스템 프롬프트**: S5의 Hard Gate + Decision Tree + Red Flags + Self-Check를 `buildCoverLetterSystemPrompt`에 통합
2. **Pre-check**: 채팅 라우트에서 메인 LLM 호출 전 사용자 메시지 분류 → 동적 힌트 주입. 규칙 기반으로 시작하고, 필요 시 경량 LLM 분류로 확장
3. **Turn 2 힌트**: 2턴 플로우(제안 → 승인) 진입 시 "saveCareerNote 호출" 힌트 자동 주입

### 한계

- **비결정성**: reasoning 모델은 같은 입력에도 다른 결과. N회 반복(≥5회)으로 통계적 평가 필요
- **시나리오 수**: 4개 시나리오는 실제 사용 패턴의 일부. 프로덕션 로그에서 추가 실패 패턴 발견 시 Red Flags에 추가 (반복 강화)
- **규칙 기반 Pre-check**: 새 패턴에는 수동 규칙 추가 필요. 프로덕션 스케일에서는 경량 LLM 분류로 전환 고려

### 범용 패턴 검증 결과

Superpowers-style enforcement + Pre-check 프레임워크의 효과가 실증됨:

```
Step 1: 벤치마크로 under-call 실패 패턴 수집          → v1 보고서
Step 2: 실패 패턴을 Red Flags로 변환                   → S5 (under-call ↓, over-call ↑)
Step 3: over-call 패턴 발견 → Pre-check 도입           → S6 (over-call ↓, 토큰 -17%)
Step 4: 시나리오 기대값 재검토 → allowed 범위 수정      → 새 경험 PASS
Step 5: 반복 (프로덕션 로그에서 새 패턴 발견 시)        → 계속
```
