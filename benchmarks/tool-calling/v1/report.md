## 1. 벤치마크 설계

### 1.1 목적

LLM이 이력서 관리 서비스의 채팅 인터페이스에서 **도구를 호출해야 할 때와 하지 말아야 할 때를 올바르게 판단**하는 능력을 측정한다. 단순한 도구 호출 성공률이 아니라, 맥락에 따른 판단력 자체를 평가하는 것이 핵심이다.

### 1.2 프롬프트 변형 (4종)

| 변형 | 설명 | 전략 |
|------|------|------|
| **S1 (최소)** | 도구 목록만 제공, 별도 지시 없음 | 모델 자체 판단력 baseline |
| **S2 (현재 프로덕션)** | 현재 운영 중인 시스템 프롬프트 | 실제 서비스 성능 측정 |
| **S3 (few-shot)** | 올바른 판단 예시를 프롬프트에 포함 | 예시 기반 학습 유도 |
| **S4 (단계별 판단)** | "새 정보인가? → 기존 기록에 있는가? → 도구가 필요한가?" 의사결정 트리 제공 | 구조화된 추론 유도 |

### 1.3 시나리오 (4종)와 2턴 플로우

| 시나리오 | 입력 | 기대 동작 | 턴 수 |
|----------|------|-----------|-------|
| **1. 새 경험** | "작년에 Rust로 고성능 메시지 큐를 만들어서 초당 50만 건 처리했어요" | saveCareerNote 제안 → 승인 후 호출 | 2턴 |
| **2. 수치 변경** | "deploy-ez Stars가 800개가 됐어요" (기존 450) | readCareerNote로 기존 기록 확인 + saveCareerNote 제안 → 승인 후 호출 | 2턴 |
| **3. 초안 요청** | "deploy-ez 중심으로 핵심 역량 문단 초안 써줘" | readDocument 호출 (참고자료 조회) | 1턴 |
| **4. 단순 질문** | "자기소개서 분량은 보통 얼마나 써야 해?" | 도구 호출 없이 텍스트 응답 | 1턴 |

**2턴 플로우의 핵심 설계**: saveCareerNote의 도구 설명에 "사용자에게 먼저 제안하고, 승인 후 호출"이라는 프로토콜이 명시되어 있다. 따라서 시나리오 1, 2는 다음과 같이 동작해야 한다:

1. **Turn 1**: 모델이 텍스트로 커리어노트 저장을 제안
2. **Turn 2**: 벤치마크가 "네, 저장해주세요" 승인 메시지를 보내고, 모델이 실제로 saveCareerNote를 호출

Turn 1에서 제안 없이 바로 도구를 호출하면 프로토콜 위반이고, 제안도 하지 않으면 도구 호출 누락으로 FAIL이다.

## 2. 프롬프트 변형별 Pass Rate 비교

### 2.1 종합 성적표

| 프롬프트 | gpt-5.4 | gpt-5.4-nano | 합산 |
|----------|---------|--------------|------|
| **S1 (최소)** | 2/4 (50%) | 1/4 (25%) | 3/8 (37.5%) |
| **S2 (현재)** | 3/4 (75%) | 1/4 (25%) | 4/8 (50%) |
| **S3 (few-shot)** | 3/4 (75%) | 1/4 (25%) | 4/8 (50%) |
| **S4 (단계별 판단)** | 3/4 (75%) | 2/4 (50%) | 5/8 (62.5%) |

### 2.2 분석

**S1 (최소)**이 가장 낮은 성적을 보인다. 도구 설명만으로는 모델이 "새 정보 → 저장 제안"이라는 맥락적 판단을 스스로 하기 어렵다. gpt-5.4조차 시나리오 1(새 경험)에서 S1으로 실패했는데, 응답을 보면 Rust 메시지 큐 경험을 자기소개서 문장에 반영하는 초안은 작성했으나, 이를 커리어노트에 저장해야 한다는 판단 자체를 하지 못했다.

**S2/S3/S4**는 gpt-5.4에서 동일하게 3/4 pass를 기록했으나, 실패 시나리오가 모두 "수치 변경"(시나리오 2)이라는 공통점이 있다. 이는 프롬프트 개선이 아니라 시나리오 자체의 난이도가 문제임을 시사한다.

**S4 (단계별 판단)**가 유일하게 gpt-5.4-nano에서 2/4를 달성하며, 소형 모델에서도 구조화된 의사결정 트리가 가장 효과적이었다. S4의 핵심은 "이 정보가 기존 기록에 있는가?"라는 중간 체크 단계를 명시적으로 요구한다는 점이다.

## 3. 모델별 차이 (gpt-5.4 vs gpt-5.4-nano)

### 3.1 전체 성적

| 지표 | gpt-5.4 | gpt-5.4-nano |
|------|---------|--------------|
| **전체 Pass Rate** | 11/16 (68.75%) | 5/16 (31.25%) |
| **제안 감지율** (FAIL 중) | 2/5 (40%) | 0/11 (0%) |
| **평균 응답 시간** | ~12.3초 | ~3.5초 |
| **평균 출력 토큰** | ~648 | ~473 |

### 3.2 결정적 차이: 제안 능력

가장 핵심적인 차이는 **제안(proposal) 능력**이다.

gpt-5.4는 FAIL한 5건 중에서도 2건(수치 변경 S3, S4)에서 텍스트로 커리어노트 저장을 제안했다. 즉, 도구 호출 판단은 했으나 readCareerNote를 먼저 호출하지 않아 FAIL이 된 것이다. 실질적으로 "제안 + 호출" 행동 자체는 수행했다.

반면 gpt-5.4-nano는 11건의 FAIL 중 **단 한 건도 텍스트 제안을 하지 않았다**. 이는 nano 모델이 "새 정보가 들어왔을 때 저장을 고려해야 한다"는 맥락적 판단 자체를 수행하지 못함을 의미한다. nano는 사용자 입력을 곧바로 자기소개서 작성 태스크로 해석하고, 도구 활용이라는 메타 레벨의 판단을 건너뛴다.

### 3.3 nano의 초안 요청 실패 패턴

특히 주목할 점은 gpt-5.4-nano가 **시나리오 3(초안 요청)에서 4/4 전패**한 것이다. gpt-5.4는 이 시나리오에서 4/4 전승했다. nano의 응답을 분석하면:

- S1: readDocument를 호출하지 않고, 시스템 프롬프트의 요약 정보만으로 바로 초안을 작성했다.
- S2: 동일하게 도구 없이 바로 초안 작성.
- S3: few-shot 예시가 있었음에도 도구 호출 없이 초안 작성.
- S4: 단계별 판단 트리가 있었음에도 도구 호출 없이 초안 작성.

nano는 "초안을 써달라"는 요청에 대해, 이미 시스템 프롬프트에 포함된 참고자료 요약만으로 충분하다고 판단한 것으로 보인다. 그러나 벤치마크의 기대 동작은 readDocument를 호출하여 원문을 확인한 후 초안을 작성하는 것이다. 이는 **hallucination 위험**과 직결되는 문제로, 요약만 보고 작성한 초안은 원문의 세부 수치나 맥락을 정확히 반영하지 못할 수 있다.

### 3.4 속도-성능 트레이드오프

gpt-5.4-nano는 평균 응답 시간이 gpt-5.4의 약 1/3.5 수준이다(~3.5초 vs ~12.3초). 도구 호출이 포함된 경우 이 차이는 더 벌어진다(2턴 완료 기준 gpt-5.4가 ~17초, nano가 ~7.7초). 그러나 빠른 응답이 잘못된 판단과 결합되면 서비스 품질에 오히려 부정적이다.

## 4. 시나리오별 난이도 분석

### 4.1 난이도 순위

| 시나리오 | gpt-5.4 Pass | nano Pass | 합산 Pass | 난이도 |
|----------|-------------|-----------|-----------|--------|
| **4. 단순 질문** | 4/4 | 4/4 | 8/8 (100%) | 매우 쉬움 |
| **3. 초안 요청** | 4/4 | 0/4 | 4/8 (50%) | 모델 의존적 |
| **1. 새 경험** | 3/4 | 1/4 | 4/8 (50%) | 어려움 |
| **2. 수치 변경** | 0/4 | 0/4 | 0/8 (0%) | 매우 어려움 |

### 4.2 시나리오 4 (단순 질문) — 완벽한 통과

두 모델 모두 4개 프롬프트 변형에서 100% 통과했다. "도구를 호출하지 않아야 하는 상황"에서의 판단은 매우 안정적이다. 응답 내용도 적절한 분량 가이드를 제공하며, 불필요한 도구 호출(false positive)이 없다.

### 4.3 시나리오 3 (초안 요청) — 모델 능력에 의존

gpt-5.4는 모든 변형에서 readDocument를 호출하여 참고자료 원문을 확인한 후 초안을 작성했다. 특히 S1(최소 프롬프트)에서도 readDocument, readCareerNote까지 총 4개 도구를 호출하며 충실하게 자료를 참조했다.

gpt-5.4-nano는 모든 변형에서 도구 호출 없이 바로 초안을 작성했다. 이는 nano가 "초안 작성에는 참고자료 조회가 선행되어야 한다"는 업무 흐름을 이해하지 못하고, 이미 가진 요약 정보로 충분하다고 판단한 것이다.

### 4.4 시나리오 1 (새 경험) — 제안 패턴의 차이

gpt-5.4는 S2/S3/S4에서 성공했으나 S1에서 실패했다. S1 실패 응답을 보면:

> "다만 현재까지 제공된 참고자료 요약에는 그 내용이 없어서, 사실관계를 더 정확히 반영하려면 먼저 관련 문서 전문을 확인한 뒤 자기소개서 문장으로 녹이는 것이 좋습니다."

흥미롭게도, "참고자료에 없는 새 정보"라는 인식은 있었지만, 이를 "커리어노트에 저장해야 한다"는 판단으로 연결하지 못했다. S2/S3/S4에서 추가된 프롬프트 지시사항이 이 연결고리를 제공한 것이다.

gpt-5.4의 S2 성공 응답에서는 마지막에 "Rust 메시지 큐 경험을 커리어노트로 남겨두자고 먼저 제안드릴 수도 있습니다"라고 제안했고, 승인 후 saveCareerNote를 호출했다. S3에서는 더 직접적으로 "이 경험, 커리어노트로도 저장해둘까요?"라고 물었다.

gpt-5.4-nano는 S4에서만 성공했다. S4 응답에서는 단계별 판단 트리의 영향으로 "원하시면, 이 경험을 커리어노트에 저장해두는 것도 제안드릴게요. 저장할까요?"라고 물었고, 이것이 제안으로 감지되어 2턴 플로우가 진행되었다.

### 4.5 시나리오 2 (수치 변경) — 전체 0% 통과, 최고 난이도

**8건 전원 FAIL**. 이 시나리오는 두 가지 동작을 동시에 요구한다:
1. readCareerNote를 호출하여 기존 기록을 확인
2. saveCareerNote를 제안하여 수치를 업데이트

#### 실패 패턴 분석

**패턴 A — 완전 누락 (6건)**: S1 프롬프트에서 양 모델 모두, 그리고 nano의 S2/S3/S4에서 도구를 전혀 호출하지 않았다. 수치 변경을 인식하고 자기소개서 문장에 반영했지만, 기존 커리어노트를 조회하거나 업데이트할 생각을 하지 않았다.

gpt-5.4-nano S1의 응답이 이 패턴을 잘 보여준다:
> "Stars 450->800처럼 수치가 업데이트된 부분을 자기소개서 문장에 반영할게요."

수치가 바뀌었다는 사실은 인지했지만, 이를 커리어노트에 반영해야 한다는 판단을 하지 못했다.

**패턴 B — 부분 호출, readCareerNote 누락 (2건)**: gpt-5.4의 S3, S4에서 saveCareerNote 제안 + 호출은 성공했지만, **readCareerNote를 먼저 호출하지 않았다**. S3 응답에서는 "관련 커리어노트가 있는지 확인하고"라고 텍스트로 언급했으나, 실제로 readCareerNote를 호출하지 않고 바로 새 노트를 생성(careerNoteId: "")했다. S4도 동일한 패턴이나, careerNoteId: "note-4"로 기존 노트 ID를 사용하여 업데이트를 시도한 점이 다르다.

**패턴 C — 조회만 하고 저장 미제안 (1건)**: gpt-5.4 S2에서는 readDocument 2건 + readCareerNote 1건을 호출하여 기존 기록을 충실히 조회했다. 그러나 이후 saveCareerNote를 제안하지 않고, 자기소개서 초안 작성으로만 이어갔다. 조회는 했지만 "수치가 바뀌었으니 기록을 업데이트해야 한다"는 최종 판단을 하지 못한 것이다.

#### 시나리오 2가 어려운 구조적 이유

이 시나리오가 어려운 이유는 **두 단계의 추론**이 필요하기 때문이다:

1. "Stars 450 -> 800"이라는 입력에서 "기존 기록된 수치가 변경됨"을 인식
2. "변경된 수치를 반영하려면 기존 기록을 먼저 조회(read)하고, 수정 후 저장(save)해야 한다"는 업무 흐름을 추론

특히 readCareerNote → saveCareerNote의 순서가 중요한데, 대부분의 모델이 이 순차적 의존관계를 놓쳤다. 시나리오 1(새 경험)은 saveCareerNote만 호출하면 되어 단순하지만, 시나리오 2는 "기존 데이터 확인 → 수정 → 저장"이라는 CRUD 패턴을 요구한다.

## 5. 최적 프롬프트 변형 추천

### 5.1 결론: S4 (단계별 판단) 추천

| 기준 | S1 | S2 | S3 | S4 |
|------|----|----|----|----|
| gpt-5.4 pass rate | 50% | 75% | 75% | 75% |
| nano pass rate | 25% | 25% | 25% | **50%** |
| 합산 pass rate | 37.5% | 50% | 50% | **62.5%** |
| 소형 모델 효과 | - | - | - | **가장 큼** |

**S4 (단계별 판단)**를 추천하는 이유:

1. **소형 모델에서 유일하게 차이를 만든다**: S1/S2/S3은 nano에서 모두 1/4인 반면, S4만 2/4를 달성했다. 향후 비용 절감을 위해 소형 모델을 활용할 가능성을 고려하면, S4가 가장 범용적이다.

2. **대형 모델에서도 동등한 성능**: gpt-5.4에서 S2/S3/S4 모두 3/4로 동일하므로, S4를 선택해도 대형 모델 성능에 손실이 없다.

3. **구조적 명확성**: 의사결정 트리를 명시하면, 모델이 "왜 이 도구를 호출해야 하는가"를 단계적으로 추론할 수 있다. S3(few-shot)의 예시 기반 학습은 "주어진 예시와 유사한 상황"에만 효과적이지만, S4의 의사결정 트리는 새로운 상황에도 일반화될 가능성이 높다.

### 5.2 한계 및 추가 개선 방향

S4도 시나리오 2(수치 변경)에서는 양 모델 모두 실패했다. 이를 개선하기 위해서는 의사결정 트리에 다음 분기를 추가하는 것이 효과적일 수 있다:

> "사용자가 기존 기록의 수치/사실이 변경되었다고 언급했는가?
> → YES: readCareerNote로 기존 기록을 먼저 확인하고, 수정된 내용으로 saveCareerNote 제안"

## 6. False Positive / False Negative 패턴 분석

### 6.1 False Negative (호출해야 하는데 안 한 경우)

전체 32건 중 17건이 FAIL이며, 모두 **false negative**(도구를 호출해야 하는데 호출하지 않은 경우)이다. False positive(도구를 호출하지 않아야 하는데 호출한 경우)는 0건이다.

#### FN 패턴 1: 자기소개서 작성에 몰입 (11건)

가장 흔한 패턴이다. 사용자의 새 정보나 수치 변경을 인식하지만, 이를 즉시 자기소개서 초안 작성에 활용하는 데 집중하며 "기록 저장"이라는 별도의 행동을 떠올리지 못한다.

대표적으로 gpt-5.4-nano의 새 경험 S2 응답:
> "Kubernetes 기반 운영/배포 자동화 스토리 흐름에, 초당 50만 건을 처리한 고성능 메시지 큐(Rust)를 성능/분산처리 역량 근거로 자연스럽게 얹을 수 있습니다."

모델은 새 경험을 자기소개서에 "자연스럽게 얹는" 것에만 관심이 있고, 이를 커리어노트에 저장하는 것은 고려하지 않았다.

#### FN 패턴 2: 정보 확인 질문으로 회피 (4건)

gpt-5.4-nano S3의 새 경험 응답이 대표적이다:
> "아래 질문에 답해주시면, Kubernetes 스토리 + Rust 메시지 큐를 한 덩어리로 연결해 핵심 역량 2번 문단을 바로 써드릴게요."

모델이 새 정보를 저장하는 대신, 사용자에게 추가 질문을 던져 더 많은 정보를 수집하려 했다. 이는 자기소개서 작성 관점에서는 합리적이지만, 커리어노트 저장이라는 도구 활용 판단은 완전히 누락되었다.

#### FN 패턴 3: 조회 후 저장 미연결 (1건)

gpt-5.4 수치 변경 S2에서 readDocument + readCareerNote를 호출하여 기존 기록을 조회했으나, "Stars 800으로 업데이트해야 한다"는 saveCareerNote 제안을 하지 않았다. 조회(read)와 수정(save)을 별개의 행동으로 인식한 것이다.

#### FN 패턴 4: 참고자료 무시 (4건)

gpt-5.4-nano의 초안 요청 4건 모두 해당. readDocument를 호출하지 않고 시스템 프롬프트의 요약만으로 초안을 작성했다. 이는 "참고자료가 있으면 원문을 확인해야 한다"는 판단을 하지 못한 것이다.

### 6.2 False Positive (호출하지 말아야 하는데 한 경우) — 0건

시나리오 4(단순 질문)에서 양 모델 모두, 모든 프롬프트 변형에서 도구를 호출하지 않았다. 이는 "도구가 불필요한 상황"에 대한 판단이 매우 안정적임을 보여준다. 프로덕션 환경에서 불필요한 도구 호출로 인한 latency 증가나 부작용 위험은 낮다.

### 6.3 부분 성공: readCareerNote 누락 패턴

gpt-5.4의 수치 변경 시나리오에서 특히 흥미로운 패턴이 발견된다:

- **S3**: 텍스트로 "관련 커리어노트가 있는지 확인하고"라고 언급 → 실제로는 readCareerNote 미호출 → 새 노트로 saveCareerNote(careerNoteId: "") 호출
- **S4**: 텍스트로 "기존 기록에 있는지 확인"을 언급 → readCareerNote 미호출 → 기존 ID로 saveCareerNote(careerNoteId: "note-4") 호출

S4에서는 careerNoteId를 기존 노트의 ID로 지정했는데, 이는 시스템 프롬프트의 요약 정보에서 note-4의 존재를 인지하고 있었기 때문이다. 그러나 readCareerNote로 현재 내용을 확인하지 않고 바로 덮어쓰기를 시도한 것은 데이터 무결성 관점에서 위험하다. 기존 내용을 읽지 않으면, 업데이트 시 기존 정보가 유실될 수 있다.

## 7. 프로덕션 시사점

### 7.1 모델 선택

**gpt-5.4를 채팅 모델로 유지하는 것이 적절하다.** 68.75% pass rate는 개선이 필요하지만, gpt-5.4-nano의 31.25%와 비교하면 2배 이상의 차이가 있다. 특히 nano는 초안 요청에서 참고자료를 조회하지 않는 치명적 문제가 있어, 이력서/자기소개서 작성이라는 핵심 기능에서 hallucination 위험이 높다.

다만 **비용 최적화가 필요한 경우**, 시나리오 유형을 사전에 분류하여 단순 질문(시나리오 4 유형)에는 nano를, 도구 호출이 필요한 시나리오에는 gpt-5.4를 사용하는 라우팅 전략을 고려할 수 있다.

### 7.2 프롬프트 개선

현재 프로덕션 프롬프트(S2)를 **S4(단계별 판단) 기반으로 전환**하되, 시나리오 2(수치 변경)에 대응하는 의사결정 분기를 추가해야 한다.

구체적으로 다음 로직을 프롬프트에 포함할 것을 제안한다:

```
도구 호출 의사결정 흐름:
1. 사용자가 새 경험/프로젝트/성과를 언급했는가?
   → YES: 커리어노트 저장을 제안 (승인 후 saveCareerNote)
2. 사용자가 기존 기록의 수치/사실 변경을 언급했는가?
   → YES: readCareerNote로 기존 기록 확인 → 수정 내용 제안 → 승인 후 saveCareerNote
3. 사용자가 문서 초안/수정을 요청했는가?
   → YES: readDocument로 관련 참고자료 조회 후 작성
4. 위 어느 것에도 해당하지 않는 일반 질문인가?
   → YES: 도구 없이 텍스트로 응답
```

### 7.3 수치 변경 시나리오 대응

시나리오 2(수치 변경)의 0% 통과율은 프로덕션에서 실제 문제가 될 수 있다. 사용자가 "Stars가 800개가 됐어요" 같은 업데이트를 알려줬을 때, 시스템이 이를 기록에 반영하지 않으면 다음 자기소개서 작성 시 구 수치(450)가 사용될 위험이 있다.

이를 해결하기 위한 추가 방안:
- 프롬프트에 "기존 수치 변경 감지 → 자동 readCareerNote" 트리거를 명시
- 시스템 프롬프트의 참고자료 요약에 수치가 포함된 경우, 사용자 입력과의 불일치를 감지하는 로직을 프롬프트 레벨에서 안내

### 7.4 2턴 플로우의 효과

2턴 플로우(제안 → 승인 → 호출)는 프로덕션에서 중요한 UX 패턴이다. 벤치마크 결과에서, gpt-5.4가 제안에 성공한 경우(5건) 모두 Turn 2에서 saveCareerNote를 올바르게 호출했다. 즉, **"제안할 수 있느냐"가 관건이지, "승인 후 호출"은 안정적**이다.

이는 프롬프트 개선의 초점이 "도구 호출 방법"이 아니라 "도구 호출이 필요한 상황 인식"에 있어야 함을 의미한다.

### 7.5 토큰 효율성

| 시나리오 유형 | gpt-5.4 평균 입력 토큰 | gpt-5.4 평균 출력 토큰 |
|--------------|----------------------|----------------------|
| 도구 호출 성공 (2턴) | ~5,600 | ~1,100 |
| 도구 호출 성공 (1턴) | ~4,400 | ~700 |
| 도구 호출 실패 | ~1,470 | ~600 |
| 단순 질문 | ~1,580 | ~270 |

도구 호출 성공 시 입력 토큰이 크게 증가하는데, 이는 도구 호출 결과(참고자료 본문 등)가 컨텍스트에 추가되기 때문이다. 2턴 플로우는 단일 턴 대비 약 25% 추가 토큰을 소모하지만, 사용자 데이터를 정확히 기록한다는 가치를 고려하면 합리적인 비용이다.
