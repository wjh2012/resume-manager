## 3. 주요 발견

### tc-2 "수치 변경" — 가장 극적인 차이

- **Tool-Calling**: 두 모델 모두 **0%** — `readCareerNote` → `saveCareerNote` ordered pair를 판단하지 못함
- **Deterministic Routing**: gpt-5.4 **90%**, nano **61%** — `saveCareerNote: true` boolean 판단만 요구하므로 성공
- **해석**: tool-calling은 2단계 순서쌍을 정확히 생성하는 것이 근본적으로 어려움. DR은 "저장 필요 여부"만 판단하면 되므로 복잡도가 낮음

### tc-3 "초안 요청" — Tool-Calling이 우세한 유일한 시나리오 (nano)

- **gpt-5.4-nano TC 59.3% vs DR 30.0%** — DR에서 `documentsToRead` 배열에 정확한 ID를 넣어야 하는 것이 난이도를 높임
- **gpt-5.4**에서는 DR 38.0% vs TC 32.0%로 DR이 약간 우세
- **해석**: 읽어야 할 문서 ID를 structured output으로 정확히 지정하는 것은 function calling에서 `readDocument` 도구를 호출하는 것과 난이도가 유사하거나 더 높음

## 6. 결론

### Deterministic Routing의 강점
1. **전체 정확도 대폭 향상**: gpt-5.4 기준 +37.6%p (79.3% vs 41.7%)
2. **비용 절반 이하**: output tokens 90% 감소로 비용 효율 극대화
3. **tc-2(수치 변경) 해결**: TC에서 0%였던 시나리오를 90%로 끌어올림
4. **md/sd 카테고리 이상 현상 해소**: TC의 25% → DR의 88~90%
5. **복잡한 페르소나에 강건**: 경력이 복잡할수록 성능이 오히려 향상

### Deterministic Routing의 약점
1. **tc-3(초안 요청) 여전히 저조**: 38% (gpt-5.4) — 문서 ID 선택은 근본적으로 어려운 과제
2. **nano의 R1(최소) 성능 불안정**: 29% — 가이드 없이는 structured output도 불안정
3. **tc-4(단순 질문)에서 TC 대비 소폭 하락**: 98% vs 100% (gpt-5.4) — nano에서는 R1이 32%로 큰 차이

### 다음 단계
1. **tc-3 개선**: `documentsToRead`에 문서 ID 대신 카테고리/키워드 기반 선택 방식 검토
2. **R2(필드별 가이드)를 기본 프롬프트로 채택**: gpt-5.4에서 90%로 최고 성능, 비용 대비 효율적
3. **nano는 R3(few-shot) 채택**: 76%로 최고 성능, 예시 의존도가 높음을 감안
4. **두 방식의 하이브리드**: 라우팅 판단은 DR로, 실제 도구 실행은 TC로 분리하는 2단계 파이프라인 검토
