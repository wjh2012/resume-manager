## 1. 테스트 개요

### 평가 기준

사용자 질문: "deploy-ez 프로젝트를 네이버 클라우드에 어필할 수 있는 부분 뽑아줘"

**정답에 가까운 행동:**
- ✅ doc-2 (포트폴리오) 전문 읽기 — deploy-ez 상세 내용 (Go, K8s, Stars 450+, npm 2K+)
- ✅ doc-3 (채용공고) 전문 읽기 — 요구사항 확인 (K8s, MSA, Go, 오픈소스)
- ❓ doc-1 (이력서) — 읽어도 되고 안 읽어도 됨
- ❓ 커리어노트 — 관련성 낮지만 참고 가능

**좋은 응답의 조건:**
- deploy-ez의 구체적 내용 언급 (Go, K8s, Stars 450+)
- 채용공고 요구사항과 매칭 (K8s 경험, 오픈소스 기여)
- 자기소개서에 쓸 수 있는 문장 형태

## 7. 최종 추천

### 가성비 최적 조합

| 순위 | 모델 + 방식 | 가격 (in/out) | 정확도 | 속도 | 근거 |
|---|---|---|---|---|---|
| 1 | **gpt-5.4-nano + 멀티스텝** | $0.20/$1.25 | ★★★★★ | ~8s | 최저가에 최고 정확도 |
| 2 | **haiku-4.5 + 분류** | $1.00/$5.00 | ★★★★☆ | ~6s | 안정적, 토큰 효율적 |
| 3 | **sonnet-4.6 + 분류** | $3.00/$15.00 | ★★★★☆ | ~14s | 응답 품질 최상급 |

### 방식 선택 가이드

```
모델이 OpenAI (GPT) 계열인가?
  → YES → 멀티스텝 사용 (분류에서 documentsToRead:[] 버그 빈발)

모델이 Anthropic (Claude) 계열인가?
  → YES → 분류방식 사용 (멀티스텝 대비 토큰 30~57% 절약, 정확도 유사)

모델이 Google (Gemini) 계열인가?
  → YES → 분류방식 사용 (멀티스텝에서 도구 호출 안 함)
```

### 핵심 인사이트

1. **"최적의 방식"은 모델에 따라 다르다** — GPT는 멀티스텝, Claude/Gemini는 분류방식이 적합
2. **GPT 분류방식의 구조적 결함** — 고성능 모델(4o, 5.4)에서 `documentsToRead:[]` 반환하는 버그가 반복됨
3. **Claude 멀티스텝의 과도한 탐색** — 문서를 너무 많이 읽어 토큰 낭비 (Sonnet 4.6 중규모: 12,958 토큰)
4. **Gemini Flash-Lite의 도구 무시** — 멀티스텝에서 도구 자체를 호출하지 않는 근본적 문제
5. **세대 간 차이는 극적** — GPT-4o → 5.4로 멀티스텝 정확도가 2/3 → 6/6으로 도약
