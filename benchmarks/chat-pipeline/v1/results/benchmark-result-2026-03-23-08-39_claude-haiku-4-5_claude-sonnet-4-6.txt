## 2. Kubernetes 기반 인프라 자동화 주도 경험

### **프로젝트: 내부 배포 자동화 도구 구축 (2022.01 ~ 2022.05)**

**역할:** DevOps/인프라 리드

#### 🎯 **Problem**
- 서비스 12개를 수동으로 배포 → 배포 시간 평균 **25분**, 휴먼 에러 빈번
- Kubernetes 클러스터 관리 표준화 부재 → 팀별로 상이한 배포 전략
- 장애 발생 시 수동 롤백으로 복구 시간 지연

#### 🔧 **Solution: Kubernetes 네이티브 GitOps 파이프라인**

**1) ArgoCD 기반 GitOps 구현**
```
GitHub (선언형 상태) → ArgoCD (자동 동기화) → Kubernetes 클러스터
```
- Git을 단일 진실 공급원(SSOT)으로 설정
- Kubernetes 리소스 변경사항 자동 반영
- **Pull Request → 승인 → 자동 배포** 워크플로우 표준화

**2) Helm Chart 표준화 및 템플릿화**
- 서비스별 중복 YAML 제거 (코드 60% 감소)
- 환경별(dev/staging/prod) Helm values 관리
- 12개 서비스를 **하나의 통합 Helm 리포지토리**로 관리

**3) 자동 롤백 시스템**
- ArgoCD Sync Wave를 활용한 무중단 배포
- 배포 실패 시 **이전 버전 자동 복구** (수동 개입 제거)
- Health Check + Readiness Probe로 안정성 강화

#### 📊 **Impact**
- **배포 시간: 25분 → 8분 (68% 단축)**
- **배포 실패율: 12% → 1.2% (휴먼 에러 제거)**
- **평균 복구 시간(MTTR): 4시간 → 15분**
- 월간 배포 횟수 5회 → 40회 증가 (배포 자신감 향상)

## 3. Kubernetes 확장성과 안정성 강화

### **프로젝트: 모니터링 및 자동 스케일링 시스템 (2021.06 ~ 2021.12)**

**역할:** 인프라 엔지니어

#### 🎯 **Kubernetes 네이티브 모니터링**

**1) Prometheus + Grafana 스택 구축**
- Kubernetes 메트릭 수집 (kube-state-metrics)
- 커스텀 메트릭 수집기 개발 (**Go 언어**)
- 서비스별 실시간 대시보드 제공

**2) SLO/SLI 기반 알림 체계**
```
SLI 정의: 가용성 99.9%, P95 응답시간 500ms
↓
AlertManager로 임계값 초과 시 자동 알림
↓
온콜(On-call) 엔지니어 자동 호출
```

**3) HPA(Horizontal Pod Autoscaler) 최적화**
- CPU/메모리 기반 자동 스케일링
- 커스텀 메트릭(요청 처리량) 기반 스케일링
- Pod 증감 예측 알고리즘으로 트래픽 변동에 선제적 대응

#### 📊 **Impact**
- **평균 응답시간: 1.2초 → 450ms (62% 개선)**
- **장애 감지 시간: 15분 → 2분 (7배 단축)**
- **MTTR: 4시간 → 45분**
- 자동 스케일링으로 **피크 타임 수용력 3배 증대**

## 4. 대규모 분산 시스템 설계 경험

### **프로젝트: Kafka 기반 이벤트 드리븐 아키텍처 (2022.06 ~ 2022.12)**

**역할:** 백엔드 개발 및 시스템 설계

#### 🎯 **Challenge**
- 모놀리식 아키텍처의 DB 병목 → 일일 500만 이벤트 처리 한계
- 서비스 간 강한 결합도 → 배포 주기 2주 (위험도 높음)
- 확장성 제한 → 신규 기능 추가 어려움

#### 🔧 **Solution: Kubernetes 위의 MSA 구축**

**1) Kafka + Kubernetes 조합**
- 토픽 설계: 도메인 이벤트 기반 (USER_REGISTERED, ORDER_CREATED 등)
- Consumer Group으로 서비스별 독립 처리 가능
- Kubernetes에서 Consumer 파드 자동 스케일링

**2) Schema Registry로 버전 관리**
- 이벤트 스키마 변경 추적
- 하위 호환성 검증 (Backward Compatibility)
- 서비스 간 메시지 형식 일관성 보장

**3) Dead Letter Queue 및 재시도 로직**
- 실패 이벤트 격리 처리
- 지수 백오프 기반 자동 재시도
- 모니터링 대시보드에서 실패율 추적

#### 📊 **Impact**
- **일일 이벤트 처리량: 500만건 안정적 처리** (zero downtime)
- **배포 주기: 2주 → 3일 (제품 출시 속도 7배 향상)**
- **서비스 간 결합도 제거** → 독립적 배포 가능
- **확장성 확보** → 새로운 Consumer 추가로 기능 확장 용이

## 성장 배경

저는 "잘 동작하는 코드"에서 나아가 **"누구나 쉽게 배포하고 운영할 수 있는 시스템"** 을 만드는 것에 관심을 가져왔습니다. 개발자라면 누구나 배포의 번거로움을 겪는다는 점에 착안하여, Kubernetes 기반 배포 자동화 CLI 도구 **deploy-ez**를 직접 설계하고 오픈소스로 공개했습니다. Go로 개발한 이 도구는 Docker 이미지 빌드부터 Kubernetes 클러스터 배포까지의 과정을 단일 명령어로 추상화한 것으로, 현재 **GitHub Stars 450+, 월 2,000+회 다운로드**를 기록하며 실제 현장에서 사용되고 있습니다. 사용자들의 피드백을 반영하며 유지보수하는 과정에서 오픈소스 기여 경험과 함께 Kubernetes의 핵심 원리를 깊이 이해할 수 있었습니다.

---

## Kubernetes 및 클라우드 인프라 역량

deploy-ez를 통한 개인적 연구에 그치지 않고, 테크스타트에서 **DevOps 리드**로서 Kubernetes를 실무에 직접 적용했습니다. GitHub Actions + ArgoCD 기반의 GitOps 파이프라인을 구축하고, **Helm Chart를 표준화하여 12개 서비스를 통합 관리**했습니다. 그 결과 평균 배포 시간을 **25분에서 8분으로 단축**했고, 배포 실패 시 이전 버전으로 자동 롤백되는 체계를 마련하여 서비스 안정성을 크게 높였습니다.

또한 Prometheus + Grafana + AlertManager 모니터링 스택을 구축하고, **Go로 커스텀 메트릭 수집기**를 직접 개발하여 SLO/SLI 기반의 알림 체계를 도입했습니다. 이를 통해 장애 평균 복구 시간(MTTR)을 **4시간에서 45분으로 단축**한 경험이 있습니다.

---

## 대규모 트래픽 처리 및 MSA 경험

테크스타트에서 서비스가 성장하며 단일 데이터베이스의 병목과 기능 간 강결합으로 인한 배포 지연 문제가 발생했습니다. 이를 해결하기 위해 **Apache Kafka 기반의 이벤트 드리븐 아키텍처**를 직접 설계하고 도입했습니다. 도메인 이벤트 기반 토픽 설계, Consumer Group으로 서비스별 독립 처리, Dead Letter Queue를 통한 실패 이벤트 관리, Schema Registry를 활용한 스키마 버전 관리까지 구현하여 **일일 500만 건의 이벤트를 안정적으로 처리**하는 시스템을 완성했습니다. 그 결과 배포 주기는 **2주에서 3일로 단축**되었고, 서비스 간 결합도가 대폭 낮아졌습니다.

---

## 신뢰할 수 있는 시스템 설계

코드랩 재직 당시 주요 PG사 장애로 인해 2시간 동안 결제가 불가능해지며 약 **500만 원의 매출 손실**이 발생하는 상황을 경험했습니다. 이후 **Circuit Breaker 패턴**을 적용하여 5초 내 3회 실패 시 보조 PG사(토스페이먼츠)로 자동 전환되는 폴백 시스템을 구축했습니다. 이후 3차례의 PG사 장애가 발생했지만 **다운타임 0초**를 유지하며 서비스 가용성을 지켰습니다. 이 경험은 저에게 회복 탄력성(Resilience) 설계가 얼마나 중요한지 체감하게 해준 전환점이었습니다.

## 2. Kubernetes 기반 배포 자동화 — deploy-ez

저는 2021년, Kubernetes 배포 과정의 복잡함을 직접 해결하고자 오픈소스 CLI 도구 **deploy-ez**를 Go로 개발했습니다.

당시 팀 내에서 Kubernetes 배포는 숙련된 DevOps 엔지니어가 아니면 쉽게 접근하기 어려운 영역이었습니다. 이 문제를 해결하기 위해 Docker 이미지 빌드부터 Kubernetes 클러스터 배포까지 **단일 커맨드로 자동화**하는 도구를 직접 설계하고 구현했습니다.

- **GitHub Stars 450+** 달성, 개발자 커뮤니티에서 실질적인 검증을 받은 오픈소스
- npm 패키지로도 배포하여 **월 2,000+ 다운로드** 기록
- Docker + Kubernetes 워크플로우를 추상화해 배포 진입 장벽을 대폭 낮춤
- Go 언어로 개발하여 경량화된 CLI 바이너리 제공

이 프로젝트는 단순한 사이드 프로젝트가 아니라, **실제 현장의 문제를 해결한 경험**이었습니다. 네이버 클라우드의 내부 개발자 도구 및 CI/CD 파이프라인 개선 업무에 직접적으로 기여할 수 있는 역량이라고 생각합니다.

## 3. 실무에서 증명한 Kubernetes 운영 경험

개인 프로젝트를 넘어, 테크스타트 재직 중 **실제 프로덕션 환경**에서 Kubernetes 기반 배포 체계를 구축한 경험이 있습니다.

사내 배포 자동화 프로젝트(2022)에서 DevOps 리드를 맡아 다음을 직접 설계하고 구현했습니다.

- **GitHub Actions + ArgoCD 기반 GitOps 파이프라인** 구축으로 코드 변경이 클러스터에 자동 반영되는 체계 완성
- **Helm chart 표준화**로 12개 서비스의 배포 설정을 통합 관리, 설정 불일치로 인한 장애 원천 차단
- 배포 시간 **평균 25분 → 8분으로 68% 단축**
- 배포 실패 시 **롤백 자동화**로 장애 영향 범위 최소화

또한 Kafka 기반 이벤트 드리븐 아키텍처 전환 과정에서 MSA 환경의 서비스 간 결합도를 낮추고, **배포 주기를 2주에서 3일로 단축**하는 데 핵심적인 역할을 했습니다. 이는 네이버 클라우드가 요구하는 **MSA 설계 및 대규모 트래픽 처리** 경험과 직결됩니다.

## 4. 안정성에 대한 집착 — 장애를 0으로 만드는 설계

클라우드 플랫폼에서 가장 중요한 것은 **안정성**이라고 생각합니다. 저는 단순히 기능을 구현하는 것을 넘어, 장애 상황을 미리 설계하는 습관을 갖고 있습니다.

결제 시스템에서는 주 PG사 장애로 2시간 결제 불가 사태를 경험한 뒤, **Circuit Breaker 패턴**과 폴백 결제 시스템을 직접 구축했습니다. 이후 동일한 장애가 3회 더 발생했지만 **다운타임 0초**를 유지했습니다.

모니터링 시스템 고도화 프로젝트에서는 Go로 **커스텀 메트릭 수집기**를 개발하고 Prometheus + Grafana + AlertManager 스택을 구축하여 **MTTR을 4시간에서 45분으로 단축**했습니다. SLO/SLI 기반 알림 체계를 도입해 팀 전체가 서비스 상태를 한눈에 파악할 수 있는 문화를 만들었습니다.
