# 도구 호출 강제 패턴 설계 (Superpowers-Style Tool-Calling Enforcement)

## 배경

### 문제

도구 호출 판단력 벤치마크 v1 결과, S4(단계별 판단)가 최고 성적(합산 62.5%)을 기록했으나 여전히 under-call이 지배적이다. 32건 중 17건 FAIL이며, 전부 false negative(호출해야 하는데 안 함). False positive는 0건.

특히 시나리오 2(수치 변경)는 0/8 전원 FAIL로, 프롬프트 변형만으로는 해결되지 않았다.

### 관찰된 실패 패턴 (벤치마크 보고서 기반)

| 패턴 | 빈도 | 설명 |
|------|------|------|
| **자기소개서 몰입** | 11건 | 새 정보를 자소서에 바로 반영하는 데 집중, 커리어노트 저장 판단을 건너뜀 |
| **질문으로 회피** | 4건 | 도구 호출 대신 사용자에게 추가 질문 |
| **참고자료 무시** | 4건 | 요약만으로 초안 작성, readDocument 미호출 |
| **조회-저장 미연결** | 1건 | readCareerNote 후 saveCareerNote를 제안하지 않음 |

### Superpowers 패턴의 통찰

[Superpowers](https://github.com/obra/superpowers)는 Claude Code에서 "LLM이 스킬을 알지만 안 쓰는" 문제를 해결했다. 핵심 메커니즘:

1. **전제 조건 (Hard Gate)**: "X를 하려면 Y가 먼저" — 금지가 아닌 선행 조건
2. **리프레이밍 (Red Flags)**: 실패 패턴을 "이런 생각이 들면 → 실제로는" 형태로 차단
3. **반복 강화**: 실패 케이스 발견 → Red Flags 추가 → 재테스트 → 수렴

이 패턴을 LLM 앱의 시스템 프롬프트 레벨에 적용한다.

## 설계

### 접근: 점진적 검증 (Prompt First → Architecture Later)

Phase 1에서 S5 프롬프트를 설계하고 벤치마크로 효과를 측정한다. under-call이 남으면 Phase 2(Pre-check), Phase 3(Post-validation)으로 아키텍처를 확장한다.

이 설계 문서는 Phase 1(S5 프롬프트)을 다룬다.

### S5 프롬프트 구조

S4의 의사결정 트리를 기반으로 Superpowers의 3가지 강제 메커니즘을 추가한다.

```
S4:
  └── 의사결정 트리 (4단계 if/else)

S5:
  ├── 1. 필수 전제 조건 (Hard Gate)
  ├── 2. 의사결정 트리 (S4 강화판)
  ├── 3. Red Flags 테이블 (리프레이밍)
  └── 4. 검증 체크리스트 (Self-Check)
```

#### 1. 필수 전제 조건 (Hard Gate)

"~하지 마세요"(금지)가 아닌 "X를 하려면 Y가 먼저"(전제 조건) 구조:

```
[필수 전제 조건]
- 초안/문단 작성은 반드시 readDocument로 원문을 확인한 후 시작하세요.
- 새 경험/수치가 언급되면 반드시 커리어노트 저장을 제안하세요.
- 기존 기록의 수치 변경 시, readCareerNote로 현재 내용을 확인한 후 saveCareerNote를 제안하세요.
```

**설계 근거**: Superpowers TDD 스킬의 "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"와 동일한 패턴. 행동을 금지하는 것이 아니라, 행동의 선행 조건을 명시한다.

**"참고자료는 요약입니다" 라인 유지**: S2~S4에 포함된 기존 안내("아래 참고자료는 요약입니다. 구체적인 경험, 수치, 세부 내용이 필요하면 readDocument 또는 readCareerNote 도구로 전문을 읽으세요")를 S5에서도 유지한다. 이 라인은 gpt-5.4의 시나리오 3(초안 요청) 4/4 통과에 기여했을 가능성이 있으며, 제거 시 회귀 위험이 있다. Hard Gate/Red Flags와 중복되지만, 기본 안내(일반적 지시)와 강제 메커니즘(구체적 전제 조건)은 역할이 다르다.

#### 2. 의사결정 트리 (S4 강화)

S4에 시나리오 2(수치 변경) 대응 분기를 명확히 분리:

```
[도구 호출 의사결정 — 위에서부터 순서대로 평가하세요]
1. 사용자가 참고자료에 이미 기록된 수치/사실의 변경을 언급했는가?
   → readCareerNote로 기존 기록을 확인하세요. 그 후 수정된 내용으로 saveCareerNote를 제안하세요.
2. 사용자가 참고자료에 없는 새 경험/성과/수치를 언급했는가?
   → saveCareerNote 저장을 제안하세요.
3. 초안/문단 작성이 필요한가?
   → readDocument로 관련 참고자료 원문을 읽은 후 작성하세요.
4. 위 어느 것에도 해당하지 않는가?
   → 도구 호출 없이 응답하세요.
```

**S4와의 차이**:
- **항목 순서**: "기존 기록 변경"(1번)을 "새 경험"(2번)보다 먼저 평가. 구체적 조건이 먼저, 일반적 조건이 나중. v1 벤치마크에서 시나리오 2가 항목 1("새 수치")에 매칭되어 readCareerNote를 건너뛰는 문제를 방지.
- **트리거 구체화**: S4의 "기존 커리어노트와 다른 정보인가?"를 "참고자료에 이미 기록된 수치/사실의 변경"으로 구체화. "다른 정보"는 모호하지만 "기록된 수치의 변경"은 명확한 트리거.
- **"참고자료에 없는"**: 2번 항목에 "참고자료에 없는"을 추가하여 1번과의 경계를 명확히 함.

#### 3. Red Flags 테이블 (리프레이밍)

벤치마크에서 발견된 4가지 실패 패턴을 직접 차단. Superpowers의 "이런 생각이 들면 → 실제로는" 구조:

```
[도구 호출 Red Flags — 이런 생각이 들면 멈추고 재확인하세요]
| 생각 | 실제 |
|------|------|
| "요약만으로 초안을 쓸 수 있다" | 요약은 개요일 뿐. 초안에는 원문의 구체적 수치와 맥락이 필요합니다. readDocument로 확인하세요. |
| "새 정보를 자기소개서에 반영하면 충분하다" | 자기소개서 반영과 커리어노트 저장은 별개입니다. 새 경험은 커리어노트 저장도 제안하세요. |
| "사용자에게 추가 질문을 먼저 하자" | 경험이 이미 구체적이면(수치, 기간, 결과 포함) 저장 제안이 우선입니다. |
| "기존 노트 ID를 알고 있으니 바로 수정하면 된다" | readCareerNote로 현재 내용을 확인해야 기존 정보 유실을 방지할 수 있습니다. |
```

**설계 근거**: Superpowers using-superpowers 스킬의 Red Flags 테이블(12행)과 동일한 패턴. LLM이 도구를 안 쓸 때 하는 "합리화"를 미리 나열하고, 각각에 대해 올바른 행동으로 리프레이밍한다.

#### 4. 검증 체크리스트 (Self-Check)

응답 직전에 전제 조건 충족 여부를 점검:

```
[응답 전 체크리스트]
□ 초안을 작성했다면 — readDocument를 호출했는가?
□ 새 경험이 언급되었다면 — 커리어노트 저장을 제안했는가?
□ 수치가 변경되었다면 — readCareerNote를 먼저 호출했는가?
하나라도 미충족이면 도구를 호출한 후 응답을 시작하세요.
```

**설계 근거**: Superpowers TDD 스킬의 8개 체크박스("Can't check all boxes? You skipped TDD. Start over.")와 동일한 패턴. 미충족 시 복구 행동("도구를 호출한 후")을 명시. 4번째 항목(false-positive 점검)은 v1에서 0건이었으나, 의사결정 트리 4번 항목과의 대칭성을 위해 포함.

### 벤치마크 v2

v1은 완결된 실험(S1~S4 비교, 보고서 완료). S5는 새로운 접근이므로 `benchmarks/tool-calling/v2/`로 분리한다.

#### 디렉토리 구조

```
benchmarks/tool-calling/
├── v1/              # S1~S4 비교 (완료, 수정하지 않음)
│   ├── prompts.ts
│   ├── scenarios.ts
│   ├── runner.ts
│   ├── openai.ts
│   ├── report.md
│   └── results/
└── v2/              # S5 강제 패턴 검증 (신규)
    ├── prompts.ts   # S4(대조군, v1에서 텍스트 복사) + S5(실험군)
    ├── runner.ts    # v2 전용 runner (v1 유틸리티 import + v2 설정)
    ├── openai.ts    # 엔트리포인트 (v1/openai.ts와 동일 구조)
    └── results/
```

#### 설계 결정

- **시나리오**: v1의 `scenarios.ts`를 직접 import (`import { SCENARIOS, ... } from '../v1/scenarios'`). 동일 시나리오로 비교해야 하므로 복제하지 않음.
- **판정 로직**: v1의 `runner.ts`에서 순수 유틸리티(`judgePass`, `detectProposal`, `extractCalls`, `sumTokens`, `runSingle`, `runWithRetry`, 타입 정의)를 import. `detectProposal`은 `runSingle` 내부에서 호출되므로 함께 export 필요. v2의 `runner.ts`에서 `runBenchmark()` 함수를 새로 작성하여 v2의 `PROMPT_VARIANTS`와 v2의 `RESULTS_DIR`을 사용. v1의 `runBenchmark()`는 `PROMPT_VARIANTS`를 내부에서 import하므로 직접 재사용 불가.
- **v1 수정 범위**: v1의 `runner.ts`에서 유틸리티 함수와 타입을 개별 export 추가. 기존 `runBenchmark` 동작에는 영향 없음.
- **S4 대조군**: v1 `prompts.ts`의 S4 프롬프트 텍스트를 v2 `prompts.ts`에 복사. 의도적 중복 — 실험 격리를 위해 v1과 v2의 프롬프트가 독립적이어야 함. v1의 S4가 나중에 변경되어도 v2 대조군에 영향 없음.
- **S5 프롬프트**: v2 `prompts.ts`에 전체 S5 프롬프트 템플릿(본 문서의 "전체 S5 프롬프트 템플릿" 섹션)을 구현. `{companyName}`, `{position}` 등은 v1과 동일하게 하드코딩 ("네이버 클라우드", "시니어 백엔드 개발자").
- **프롬프트 변형**: S4(대조군)와 S5(실험군) 2개만 비교. S1~S3은 v1에서 이미 열등함이 입증됨.
- **결과 디렉토리**: `v2/results/`에 별도 저장.
- **openai.ts**: v1의 `openai.ts`와 동일 구조 (모델 생성 → `runBenchmark(modelConfig)` 호출). v2의 `runBenchmark`를 import.

#### 측정 기준

- **1차 지표**: S5 vs S4 pass rate 비교 (전체, 시나리오별, 모델별)
- **핵심 관심사**: 시나리오 2(수치 변경)에서 개선 여부 (현재 0/8)
- **모델**: gpt-5.4, gpt-5.4-nano
- **실행 규모**: 2변형 × 4시나리오 × 2모델 = 16 runs

#### 성공 기준

| 지표 | 현재 (S4) | 목표 (S5) |
|------|-----------|-----------|
| gpt-5.4 전체 pass rate | 75% (3/4) | 100% (4/4) |
| gpt-5.4-nano 전체 pass rate | 50% (2/4) | 75% (3/4) |
| 시나리오 2 pass rate (gpt-5.4) | 0% (0/1) | 1/1 |
| 시나리오 2 pass rate (nano) | 0% (0/1) | ≥0/1 (개선 확인) |

### 후속 단계 (Phase 2, 3 — 본 설계 범위 밖)

S5로 목표 미달 시:

- **Phase 2 — Pre-check**: 메인 LLM 호출 전, 경량 분류로 도구 필요 여부를 판단하여 시스템 프롬프트에 동적 힌트 주입
- **Phase 3 — Post-validation**: LLM 응답 후, 도구 호출 누락 검증 → 미충족 시 강화 힌트로 재시도 (최대 1회)

## 범용 패턴: Superpowers-Style Tool-Calling Enforcement

이 설계에서 추출한 범용 패턴. 어떤 LLM 앱이든 tool-calling under-call 문제에 적용 가능.

### 프레임워크

```
1. Hard Gate     — "X를 하려면 Y가 먼저" 형태의 전제 조건
2. Decision Tree — 구조화된 if/else 의사결정 흐름
3. Red Flags     — 실제 실패 패턴에서 추출한 "생각 → 실제" 리프레이밍 테이블
4. Self-Check    — 응답 전 전제 조건 충족 여부 점검
```

### 적용 방법

```
Step 1: 벤치마크로 under-call 실패 패턴 수집
Step 2: 실패 패턴을 Red Flags로 변환 ("생각 → 실제" 쌍)
Step 3: Hard Gate + Decision Tree + Red Flags + Self-Check를 시스템 프롬프트에 통합
Step 4: 벤치마크 재실행으로 효과 측정
Step 5: 새 실패 패턴 발견 시 Red Flags에 추가 → Step 4로 반복
```

### 핵심 원칙

- **금지가 아닌 전제 조건**: "~하지 마세요"는 LLM 안티패턴. "X를 하려면 Y가 먼저"로 표현
- **데이터 기반 Red Flags**: 추측이 아닌 실제 벤치마크 실패에서 추출
- **반복 강화**: Red Flags는 고정이 아닌 진화하는 목록. 새 실패 패턴 발견 시 계속 추가
- **측정 가능한 효과**: 모든 변경은 벤치마크 pass rate로 검증
- **구체적 조건이 먼저**: Decision Tree의 항목은 구체적(specific) 조건이 일반적(general) 조건보다 위에 와야 한다. LLM은 위에서부터 순서대로 평가하므로, 일반적 조건이 먼저 오면 구체적 조건에 도달하기 전에 매칭된다.
