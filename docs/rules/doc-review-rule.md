# 문서 점검 규칙

## 보고 전 검증 원칙

문서 간 모순·중복·개선사항을 보고하기 전에 반드시 실제 상태를 검증한다.

### 1. 존재 여부 검증

파일·디렉토리·브랜치가 "없다"고 보고하기 전에 `git branch -a`, `ls`, `Glob` 등으로 실제 확인한다.

### 2. 맥락 구분

두 문서가 다른 내용을 말할 때, 스코프(용도·대상·시점)가 다른 건 아닌지 확인한다. 스코프가 다르면 모순이 아니다.

- 예: "코드 리뷰 전 spec-deviations 참조"와 "구현 전 references 전체 참조"는 시점과 목적이 다름

### 3. 보완 관계 확인

한쪽에 없는 정보가 다른 문서에서 설명되어 있으면 모순이 아니라 역할 분담이다.

- 예: features 문서에 카카오 로그인이 나열되어 있고, GitHub Issues에 비활성화 사유가 기재되어 있으면 정상

### 4. 보고 전 자문

보고 목록 최종 제출 전 각 항목에 대해 "이걸 실제로 확인했나?" 점검한다.
